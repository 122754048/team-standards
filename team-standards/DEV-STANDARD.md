# 开发规范 v1.0.0

> 适用岗位：客户端开发（小前）、服务端开发（小后）
> 制定日期：2026-03-23
> 状态：生效中

---

## 一、核心职责

开发工程师是功能实现和代码质量的第一责任人，负责：
- 根据 PRD + 设计稿进行编码实现
- 保证代码质量、可读性、可维护性
- 自测通过后提交测试
- 持续集成和部署

## 二、工作流程

### 2.1 接收任务 → 技术方案

**输入：**
- PRD：`shared-projects/<项目名>/PRD.md`
- 设计稿：`shared-projects/<项目名>/DESIGN.md` 和 `src/` 目录

**必须输出：**
1. 技术选型（框架、语言、主要依赖库，有合理理由）
2. 目录结构规划
3. 核心模块划分
4. 关键技术难点和应对方案
5. 预估工时

**禁止：**
- ❌ 不看 PRD 和设计稿就开始写代码
- ❌ 技术选型无理由，跟风选择
- ❌ 不规划目录结构就开始堆代码

### 2.2 编码规范

#### 2.2.1 通用规则

| 规则 | 要求 | 禁止 |
|---|---|---|
| 命名 | 变量名、函数名、文件名有意义，见名知意 | 禁止 a、b、c、temp1、data2 |
| 注释 | 关键逻辑必须有注释，复杂函数必须有 JSDoc/docstring | 禁止无注释的复杂逻辑 |
| 错误处理 | 所有外部调用、文件操作、网络请求必须有错误处理 | 禁止裸跑，禁止空 catch |
| 代码风格 | 同一项目统一代码风格，使用 ESLint/Prettier 或等效工具 | 禁止风格混乱 |
| 文件长度 | 单文件不超过 300 行，超过必须拆分 | 禁止上帝文件 |
| 函数长度 | 单函数不超过 50 行，超过必须拆分 | 禁止上帝函数 |

#### 2.2.2 目录结构标准

```
shared-projects/<项目名>/
├── README.md              ← 项目说明、如何运行、技术栈、环境要求
├── CHANGELOG.md           ← 变更记录
├── PRD.md                 ← 需求文档
├── DESIGN.md              ← 设计文档
├── package.json / pom.xml  ← 项目配置（根据语言）
├── .gitignore             ← Git 忽略规则
├── src/                   ← 源代码
│   ├── index.html / main.js  ← 入口文件
│   ├── components/         ← 组件（Web）
│   ├── pages/              ← 页面（Web）
│   ├── api/                ← API 接口
│   ├── utils/              ← 工具函数
│   ├── styles/             ← 样式文件
│   └── assets/             ← 静态资源
├── tests/                 ← 测试代码
├── docs/                  ← 项目文档
└── scripts/               ← 构建/部署脚本
```

#### 2.2.3 Git 规范

**分支策略：**

| 分支类型 | 命名格式 | 用途 |
|---|---|---|
| main | `main` | 稳定发布分支 |
| develop | `develop` | 开发集成分支 |
| feature | `feature/<功能名>` | 新功能开发 |
| fix | `fix/<问题描述>` | Bug 修复 |
| refactor | `refactor/<描述>` | 重构 |

**Commit 规范：**
```
feat: 新增用户登录功能
fix: 修复表单验证不生效的问题
refactor: 重构 API 请求层
docs: 更新 README 部署说明
chore: 更新依赖版本
test: 新增登录模块单元测试
```

**禁止：**
- ❌ 直接在 main 分支开发
- ❌ commit message 写"update"、"fix bug"、"修改"
- ❌ 一次性提交大量不相关的修改

### 2.3 自测要求

开发完成后，提交测试前，**必须**完成以下自检：

- [ ] 代码可正常运行（无编译错误、无运行时崩溃）
- [ ] 主要功能流程跑通（按 PRD 的验收标准逐条验证）
- [ ] 页面/接口在主流浏览器/环境下正常
- [ ] 无明显的 UI 错位、样式问题
- [ ] 无 console.error / 未处理的异常
- [ ] 代码已 commit 并 push 到对应分支

**禁止：**
- ❌ 代码写完不跑就提交测试
- ❌ 明知有 bug 先提交再说

### 2.4 实时保存原则

- 每完成一个**有意义的阶段性工作**，立即 `git add && git commit && git push`
- 禁止等到"全部完成"再提交
- 大重构前先 commit 当前状态作为回退点

## 三、开发工具要求

| 工具 | 用途 | 必须使用 |
|---|---|---|
| Git | 版本管理 | ✅ |
| 代码格式化工具 | ESLint/Prettier（前端）或等效 | ✅ |
| 包管理器 | npm/yarn/pnpm 或等效 | ✅ |
| 构建工具 | Vite/Webpack/等（根据项目） | ✅ 如需构建 |
| 开发服务器 | 本地可运行 | ✅ |

## 四、输出物清单

| 输出物 | 存放位置 | 格式 | 时机 |
|---|---|---|---|
| 技术方案 | 通过 sessions_send 发给 PM | 结构化文字 | 开发前 |
| 源代码 | `shared-projects/<项目名>/src/` | 代码文件 | 开发中 |
| README | `shared-projects/<项目名>/README.md` | Markdown | 项目初始化时 |
| CHANGELOG | `shared-projects/<项目名>/CHANGELOG.md` | Markdown | 每次提交后更新 |
| 构建产物 | `shared-projects/<项目名>/dist/` 或部署地址 | 构建文件/URL | 开发完成后 |

## 五、验收标准

开发产出物符合以下标准才算通过：
- [ ] 代码可直接 clone 并运行
- [ ] 目录结构符合规范
- [ ] 命名有意义，注释充分
- [ ] 错误处理完善
- [ ] PRD 中 P0 功能全部实现
- [ ] 自检清单全部通过
- [ ] Git 历史清晰，commit message 规范

## 六、红线

以下行为直接打回：

1. **代码无法运行** — clone 后无法启动
2. **无错误处理** — 关键路径没有 try-catch 或等效处理
3. **上帝文件** — 单文件超过 500 行不拆分
4. **提交空壳** — 只有 TODO/FIXME/占位符，没有实际实现
5. **不保存就断** — 未 commit 就结束工作
6. **未自测就交** — 明知有问题还提交测试
