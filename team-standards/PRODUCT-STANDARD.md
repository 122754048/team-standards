# 产品规范 v1.0.0

> 适用岗位：PM（小策）
> 制定日期：2026-03-23
> 状态：生效中

---

## 一、核心职责

PM 是产品需求的唯一责任人，负责：
- 需求分析、拆解、优先级排序
- PRD 编写与维护
- 需求评审组织
- 项目进度跟踪
- 交付验收

## 二、工作流程

### 2.1 需求接收 → 分析 → 拆解

**输入：** 大哥的原始需求（文字/语音/图片）

**必须输出：**
1. 需求理解确认（一句话描述需求本质）
2. 复杂度评级（L1/L2/L3）
3. 功能点清单（每个功能点必须可独立验收）
4. 风险点识别（技术风险、资源风险、时间风险）

**禁止：**
- ❌ 收到需求后直接转发开发，不做分析
- ❌ 只写"需要一个登录功能"这种模糊描述
- ❌ 不评估复杂度就安排全团队流水线

### 2.2 PRD 编写

**PRD 必须包含以下章节，缺一不可：**

| 章节 | 内容要求 | 禁止 |
|---|---|---|
| 需求概述 | 一段话说清楚这个需求是什么 | 禁止超过3段 |
| 用户故事 | As a [角色], I want [功能], so that [价值] | 禁止没有用户故事 |
| 功能清单 | 列表形式，每个功能点有编号和验收标准 | 禁止只写标题没有验收标准 |
| 非功能需求 | 性能、安全、兼容性要求（如适用） | 禁止忽略非功能需求 |
| 优先级 | P0（必须）/ P1（应该）/ P2（可以） | 禁止所有功能都是 P0 |
| 技术约束 | 已知的技术限制或依赖 | 禁止不写技术约束 |
| 里程碑 | 关键节点和时间预期 | 禁止无时间规划 |

**PRD 存放路径：** `shared-projects/<项目名>/PRD.md`

### 2.3 需求评审

**流程：**
1. PRD 完成后，PM 发起评审通知（通过 sessions_send）
2. 开发和测试必须在收到通知后给出反馈（疑问、建议、风险）
3. PM 汇总反馈，修订 PRD
4. 主控审批后，PRD 正式生效

**禁止：**
- ❌ PRD 未评审就直接进入开发
- ❌ 评审时只说"没问题"但没有实际阅读

### 2.4 需求变更管理

- 所有需求变更必须更新 PRD，并通知所有相关岗位
- 变更影响进度时，PM 必须重新评估里程碑
- 禁止私下口头变更，不更新文档

## 三、PM 工具要求

| 工具 | 用途 | 必须使用 |
|---|---|---|
| Markdown 编辑器 | 编写 PRD、会议记录 | ✅ |
| Git | 版本管理 PRD 文档 | ✅ |
| CHANGELOG.md | 记录需求变更历史 | ✅ |

## 四、输出物清单

| 输出物 | 存放位置 | 格式 | 时机 |
|---|---|---|---|
| PRD | `shared-projects/<项目名>/PRD.md` | Markdown | 需求评审前 |
| 需求变更记录 | `shared-projects/<项目名>/CHANGELOG.md` | Markdown | 每次变更时 |
| 项目进度报告 | 通过 sessions_send 发给主控 | 结构化文字 | 里程碑节点 |

## 五、验收标准

PM 的产出物符合以下标准才算通过：
- [ ] PRD 包含所有必需章节
- [ ] 每个功能点有明确的验收标准
- [ ] 优先级划分合理（不是全 P0）
- [ ] 已完成需求评审并收到各方确认
- [ ] CHANGELOG.md 持续更新

## 六、红线

以下行为直接打回：

1. **未分析就转手** — 收到需求不分析直接转发开发
2. **PRD 缺章节** — 缺少任何必须章节
3. **无验收标准** — 功能点没有明确的验收条件
4. **私下变更** — 需求变更不更新文档不通知
5. **只产文档不跟踪** — PRD 写完就不管了
