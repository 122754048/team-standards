# 测试规范 v1.0.0

> 适用岗位：测试工程师（小测）
> 制定日期：2026-03-23
> 状态：生效中

---

## 一、核心职责

测试工程师是质量保障的唯一责任人，负责：
- 根据 PRD 编写测试用例
- 执行功能测试、回归测试
- 记录和管理 Bug
- 验证 Bug 修复
- 出具测试报告

## 二、工作流程

### 2.1 接收任务 → 测试分析

**输入：**
- PRD：`shared-projects/<项目名>/PRD.md`
- 设计稿：`shared-projects/<项目名>/DESIGN.md`
- 开发完成通知（通过 sessions_send）

**必须输出：**
1. 测试范围（哪些功能在本次测试范围内）
2. 测试环境说明（浏览器/设备/系统版本）
3. 测试用例清单
4. 预计测试时间

**禁止：**
- ❌ 不看 PRD 就开始测试
- ❌ 只凭感觉测试，没有测试用例
- ❌ 遗漏 PRD 中的 P0 功能

### 2.2 测试用例编写

**每个测试用例必须包含：**

| 字段 | 要求 | 禁止 |
|---|---|---|
| 用例编号 | 唯一标识，如 TC-001 | 禁止无编号 |
| 用例标题 | 一句话说清测试什么 | 禁止模糊描述 |
| 前置条件 | 执行前需要满足的条件 | 禁止忽略前置条件 |
| 测试步骤 | 逐步操作，1-2-3-... | 禁止一步到位 |
| 预期结果 | 每个步骤对应的预期表现 | 禁止"应该正常" |
| 优先级 | P0/P1/P2 | 禁止无优先级 |

**测试用例模板：**

```markdown
### TC-001: 用户登录 - 正确账号密码

- **前置条件：** 用户已注册，账号 test@example.com，密码 Test@123
- **优先级：** P0
- **测试步骤：**
  1. 打开登录页面
  2. 输入账号 test@example.com
  3. 输入密码 Test@123
  4. 点击登录按钮
- **预期结果：**
  1. 登录按钮显示加载状态
  2. 登录成功，跳转到首页
  3. 页面显示用户昵称
```

**禁止：**
- ❌ 测试用例只写标题，没有步骤和预期结果
- ❌ 预期结果写"正常工作"、"没问题"
- ❌ 照搬 PRD 原文作为测试用例

### 2.3 Bug 管理规范

**Bug 报告必须包含：**

| 字段 | 要求 | 禁止 |
|---|---|---|
| Bug 编号 | 唯一标识，如 BUG-001 | 禁止无编号 |
| 标题 | 简明扼要描述问题 | 禁止"有bug" |
| 严重程度 | Critical / Major / Minor / Trivial | 禁止无严重程度 |
| 复现步骤 | 逐步操作，必须可复现 | 禁止"有时候会出" |
| 预期结果 | 应该是什么样 | 禁止缺失 |
| 实际结果 | 实际是什么样 | 禁止缺失 |
| 环境信息 | 浏览器/设备/系统版本 | 禁止缺失 |

**严重程度定义：**

| 级别 | 定义 | 示例 |
|---|---|---|
| **Critical** | 核心功能无法使用，或数据丢失 | 登录后白屏、支付失败、数据删除丢失 |
| **Major** | 功能异常但有绕过方式 | 表单验证不生效但可以手动避免 |
| **Minor** | UI 错位、文案错误等 | 按钮位置偏移 10px |
| **Trivial** | 拼写错误、颜色色差等 | 说明文字少了一个句号 |

### 2.4 测试执行

**测试流程：**

1. 准备测试环境
2. 按优先级执行测试用例：P0 → P1 → P2
3. 记录每个用例的执行结果（通过/失败/阻塞）
4. 发现 Bug 立即记录，通过 sessions_send 通知开发
5. 所有 P0 用例通过后，验证修复的 Bug
6. 输出测试报告

**禁止：**
- ❌ 不按用例执行，随机点一点
- ❌ 发现 Bug 不记录不通知
- ❌ 修改 Bug 后不复测就说修好了

### 2.5 回归测试

以下情况必须执行回归测试：
- 新功能上线前
- Bug 修复后
- 影响核心流程的代码变更后

回归测试范围：
- 至少覆盖所有 P0 测试用例
- 覆盖与变更相关的 P1 用例

## 三、测试工具要求

| 工具 | 用途 | 必须使用 |
|---|---|---|
| 测试用例文档 | 编写和管理测试用例 | ✅ |
| 浏览器 DevTools | 前端调试、网络请求检查 | ✅ Web 项目 |
| 测试报告模板 | 输出标准化测试报告 | ✅ |

## 四、输出物清单

| 输出物 | 存放位置 | 格式 | 时机 |
|---|---|---|---|
| 测试用例 | `shared-projects/<项目名>/tests/test-cases.md` | Markdown | 测试开始前 |
| Bug 清单 | `shared-projects/<项目名>/TEST-REPORT.md` | Markdown | 测试过程中持续更新 |
| 测试报告 | `shared-projects/<项目名>/TEST-REPORT.md` | Markdown | 测试完成后 |

## 五、测试报告模板

```markdown
# 测试报告 — <项目名>

## 基本信息

- **测试日期：** YYYY-MM-DD
- **测试人员：** 小测
- **测试范围：** [列出本次测试的功能模块]
- **测试环境：** [浏览器/设备/系统版本]

## 测试结果概览

| 指标 | 数值 |
|---|---|
| 用例总数 | X |
| 通过 | X |
| 失败 | X |
| 阻塞 | X |
| 通过率 | X% |

## Bug 统计

| 严重程度 | 数量 | 状态 |
|---|---|---|
| Critical | X | 已修复 X / 未修复 X |
| Major | X | 已修复 X / 未修复 X |
| Minor | X | 已修复 X / 未修复 X |
| Trivial | X | 已修复 X / 未修复 X |

## 结论

- [ ] ✅ 通过 — 所有 P0 用例通过，无 Critical/Major 未修复 Bug
- [ ] ⚠️ 有条件通过 — 有 Minor/Trivial 未修复 Bug，不影响核心功能
- [ ] ❌ 不通过 — 存在 Critical/Major 未修复 Bug 或 P0 用例失败

## 建议

[对下一步的建议]
```

## 六、验收标准

测试产出物符合以下标准才算通过：
- [ ] 测试用例覆盖 PRD 中所有 P0 功能
- [ ] 每个测试用例有完整的步骤和预期结果
- [ ] Bug 报告可复现、信息完整
- [ ] 测试报告包含数据、结论、建议
- [ ] Critical 和 Major Bug 验证修复

## 七、红线

以下行为直接打回：

1. **无用例就测试** — 没有测试用例就开始测试
2. **模糊预期** — 预期结果写"正常"、"没问题"
3. **Bug 不可复现** — Bug 报告没有复现步骤或无法复现
4. **不复测就通过** — 开发说修了就信了，不复测
5. **只报 Bug 不跟踪** — 报了 Bug 不管后续，不验证修复
6. **测试报告只有文字** — 没有数据统计，没有明确的通过/不通过结论
