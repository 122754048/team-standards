# 设计规范 v1.0.0

> 适用岗位：UI 设计师（小艺）
> 制定日期：2026-03-23
> 状态：生效中

---

## 一、核心职责

UI 设计师是视觉和交互体验的唯一责任人，负责：
- 理解 PRD 中的功能需求和用户故事
- 设计页面布局、视觉风格、交互流程
- 输出可交付的设计稿和设计资源
- 与开发对齐设计实现细节

## 二、工作流程

### 2.1 接收需求 → 设计分析

**输入：** PRD（`shared-projects/<项目名>/PRD.md`）

**必须输出：**
1. 设计目标（一句话说明本次设计的核心体验目标）
2. 页面清单（每个页面的用途和关键元素）
3. 交互流程图（页面之间的跳转关系）
4. 风格定义（色彩、字体、间距、圆角等基础变量）

**禁止：**
- ❌ 不看 PRD 直接开始画图
- ❌ 只输出视觉稿不输出交互说明
- ❌ 设计与 PRD 功能需求脱节

### 2.2 设计稿输出

**设计稿必须包含：**

| 内容 | 要求 | 禁止 |
|---|---|---|
| 页面布局 | 清晰的元素排列，标注关键间距 | 禁止"大概放这里" |
| 交互说明 | 每个可交互元素的状态（正常/悬停/点击/禁用/加载/错误） | 禁止只有静态图 |
| 响应式方案 | 不同屏幕尺寸的适配方案（如适用） | 禁止只出一套尺寸 |
| 设计变量 | 颜色值、字体大小、间距值必须精确标注 | 禁止"差不多就行" |
| 资源清单 | 所需图标、图片、插图清单 | 禁止遗漏资源 |

**设计稿存放路径：**
- 设计文档：`shared-projects/<项目名>/DESIGN.md`
- 设计资源：`shared-projects/<项目名>/src/assets/`
- 样式代码：`shared-projects/<项目名>/src/styles/`（如项目为 Web）

### 2.3 设计评审

**流程：**
1. 设计稿完成后，通知开发和 PM 评审
2. 开发确认可实现性（技术可行性、工时评估）
3. PM 确认符合需求意图
4. 修订后进入开发阶段

**禁止：**
- ❌ 设计稿未评审就交给开发
- ❌ 开发反馈不可实现后不改设计硬推

### 2.4 设计交付标准

设计稿交付给开发时，必须满足：
- 所有尺寸、颜色、字体有精确数值
- 交互状态完整定义
- 资源文件已导出并放在指定目录
- 设计变量已整理为 CSS 变量或等效格式（Web 项目）

## 三、设计工具要求

| 工具 | 用途 | 必须使用 |
|---|---|---|
| 代码生成（Codex + Pencil） | 生成 UI 原型和页面代码 | ✅ Web 项目 |
| CSS 变量体系 | 统一设计变量管理 | ✅ Web 项目 |
| SVG 工具 | 图标和矢量图形制作 | ✅ 如需自定义图标 |

**禁止使用纯文字描述代替设计稿。** 设计是视觉工作，必须输出可视化内容。

## 四、输出物清单

| 输出物 | 存放位置 | 格式 | 时机 |
|---|---|---|---|
| 设计文档 | `shared-projects/<项目名>/DESIGN.md` | Markdown | 设计评审前 |
| 页面代码（Web） | `shared-projects/<项目名>/src/` | HTML/CSS/JS | 设计通过后 |
| 样式变量 | `shared-projects/<项目名>/src/styles/variables.css` | CSS | 设计通过后 |
| 图标/图片资源 | `shared-projects/<项目名>/src/assets/` | SVG/PNG/WebP | 设计通过后 |

## 五、验收标准

设计产出物符合以下标准才算通过：
- [ ] 设计文档包含所有必须内容
- [ ] 页面布局清晰，关键尺寸有标注
- [ ] 交互状态完整（至少覆盖：正常、悬停、点击、禁用）
- [ ] 设计变量（颜色、字体、间距）有精确数值
- [ ] 代码可直接运行，不是占位符
- [ ] 资源文件已导出且格式正确

## 六、红线

以下行为直接打回：

1. **纯文档无产出** — 只写了设计说明文档，没有实际的页面/代码/设计稿
2. **设计与需求脱节** — 设计稿与 PRD 功能需求不一致
3. **无精确标注** — 尺寸、颜色、字体用"差不多"描述
4. **缺少交互说明** — 只有静态页面，没有交互状态定义
5. **不可实现设计** — 技术上无法实现或成本过高，且未沟通调整
